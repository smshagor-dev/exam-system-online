## Top Findings

### 1. Production build blocker in rich text editor

Severity: critical

- [src/components/editor/RichTextEditor.tsx](/D:/Public/exam-system-online/src/components/editor/RichTextEditor.tsx:165)
- [src/components/editor/RichTextEditor.tsx](/D:/Public/exam-system-online/src/components/editor/RichTextEditor.tsx:186)

The `heading.options` array is inferred as generic strings, but CKEditor expects stricter heading model literals such as `heading1` through `heading6`. Because of that mismatch, the build fails at TypeScript validation.

Recommended action:

- explicitly type the editor config as CKEditor `EditorConfig`
- type `heading.options` with the CKEditor heading option type or cast the literals narrowly

### 2. Student exam attempt page has hook-ordering and runtime-risk issues

Severity: critical

- [src/app/student/exams/[id]/attempt/page.tsx](/D:/Public/exam-system-online/src/app/student/exams/[id]/attempt/page.tsx:170)
- [src/app/student/exams/[id]/attempt/page.tsx](/D:/Public/exam-system-online/src/app/student/exams/[id]/attempt/page.tsx:219)

`startAutoSave` is called before its declaration. ESLint flags this as an immutability/hook correctness problem, and in a high-pressure exam flow this is not a cosmetic issue. This page is directly tied to live exam attempts, timer handling, auto-save, and auto-submit behavior.

Risk:

- unstable client behavior during exam sessions
- answer persistence bugs
- countdown and submission timing regressions

Recommended action:

- move helper declarations above the effect that uses them, or convert them to stable callbacks
- retest live attempt flow end to end with Socket.IO

### 3. Navigation shells rely on setState inside route-change effects

Severity: high

- [src/components/admin/AdminShell.tsx](/D:/Public/exam-system-online/src/components/admin/AdminShell.tsx:99)
- [src/components/student/StudentShell.tsx](/D:/Public/exam-system-online/src/components/student/StudentShell.tsx:49)

Multiple shell components close sidebar/profile state inside effects that run on route changes. React’s lint rules flag these as cascading render risks.

Risk:

- unnecessary re-renders on every navigation
- brittle shell behavior as UI complexity grows

Recommended action:

- derive route-based UI state where possible
- use event handlers or route comparisons during render instead of immediate effect-driven state resets

### 4. Login and verification flows contain effect-driven state writes

Severity: high

- [src/app/(auth)/login/page.tsx](/D:/Public/exam-system-online/src/app/(auth)/login/page.tsx:53)
- [src/app/(auth)/verify-account/page.tsx](/D:/Public/exam-system-online/src/app/(auth)/verify-account/page.tsx:39)
- [src/components/i18n/LanguagePreferenceModal.tsx](/D:/Public/exam-system-online/src/components/i18n/LanguagePreferenceModal.tsx:11)

Auth and language preference flows depend on effect bodies that synchronously update component state or mutate document state in ways the current lint profile rejects.

Risk:

- fragile first-load behavior
- hard-to-debug user onboarding issues
- hydration and navigation regressions

Recommended action:

- initialize from URL/storage using state initializers where possible
- isolate document and cookie side effects into event handlers or guarded utility functions

### 5. Server entrypoint is operationally important but lint-noisy

Severity: medium

- [server.js](/D:/Public/exam-system-online/server.js:12)
- [server.js](/D:/Public/exam-system-online/server.js:36)
- [server.js](/D:/Public/exam-system-online/server.js:103)

The runtime boot path uses CommonJS `require()` heavily, automatically runs `prisma db push` unless `SKIP_DB_MIGRATIONS=true`, and starts the academic promotion scheduler on server boot.

Risk:

- mixed lint signal because server runtime is intentionally CommonJS
- accidental schema changes during startup in the wrong environment
- background job behavior may be overlooked during deploys

Recommended action:

- exclude deliberate Node runtime files from client-focused lint rules, or migrate them intentionally
- make environment-specific DB boot behavior explicit in deployment docs and scripts

### 6. `.env.example` currently exposes a real-looking database URL

Severity: high

- [.env.example](/D:/Public/exam-system-online/.env.example:6)

The sample env file includes a populated MongoDB connection string instead of a placeholder.

Risk:

- credential leakage
- accidental reuse across environments
- poor secret hygiene in handoffs

Recommended action:

- replace with a sanitized placeholder immediately
- rotate the referenced credential if it has ever been valid
